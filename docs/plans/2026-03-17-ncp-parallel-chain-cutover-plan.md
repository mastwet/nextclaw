# NCP 并行链路切换方案（Phase 1）

## 1. 背景与决策

当前我们不是采用“在旧链路上持续小步适配替换”的方式推进 NCP，而是采用更稳妥的双链路策略：

- 在保持底层存储层一致的前提下，额外构建一套新的 NCP 前后端链路
- 新旧链路代码明确分离，避免边改边污染
- 最终切换点放在前端入口层
- 验证稳定后，一次性下线旧链路
- 若验证失败，可直接删除新链路并回到旧链路

该策略的核心目标是：最大化可回滚性与可控性，同时保证架构边界清晰。

## 2. 目标与非目标

### 2.1 目标

1. 在不改动底层存储事实模型的前提下，构建完整 NCP 新链路（backend + frontend）。
2. 新旧链路实现物理隔离、职责隔离、依赖隔离，避免“逻辑共存但实际耦合”。
3. 将切换控制集中在前端入口，支持快速切换和回滚。
4. 为最终“删除旧链路”提前设计退出条件，避免长期双实现。

### 2.2 非目标

1. 第一阶段不迁移底层存储层本体，仅做适配。
2. 第一阶段不追求所有页面和边缘功能同步迁移，只覆盖 agent 核心主链路。
3. 第一阶段不允许新链路反向依赖旧链路业务逻辑。

## 3. 核心策略：双链路并行 + 前端切换

## 3.1 双链路模型

- Legacy Chain（旧链路）：
  - 保持现有 NextClaw agent 前后端实现不动，作为稳定基线。
- NCP Chain（新链路）：
  - 使用 NCP 协议、runtime、state、transport、react binding 重新组装完整链路。
  - 通过 storage adapter 对接现有存储层，保证数据一致。

两条链路在代码上必须明确分开，禁止共享业务编排代码；可共享的仅限 truly-generic 的底层工具与类型。

## 3.2 切换点设计

统一在前端入口层设置“链路选择开关”，由该开关决定：

- 调用哪个 API host/path（legacy vs ncp）
- 使用哪套 runtime/controller/container 组合
- 渲染哪套页面容器（或同页面下不同 provider 注入）

后端不作为切换主入口，后端仅提供两套路由/服务并行可用。

## 3.4 前端代码分离（硬约束）

前端必须做到“代码分离”，不只是运行时分支：

1. Legacy 与 NCP 各自拥有独立容器层与编排层，禁止在同一个容器文件内用大量 `if/else` 混写双链路逻辑。
2. 链路开关只存在于入口装配层（router/page entry/provider entry），不下沉到通用展示组件。
3. `@nextclaw/agent-chat-ui` 作为共享展示层基座，Legacy 与 NCP 默认共用这套 UI 组件、view-model 契约和 UI-owned hooks。
4. NCP 前端链路不得反向依赖 legacy 业务模块；若存在依赖，视为未完成分离。
5. 双链路都应先把各自运行时数据适配成统一展示契约，再进入共享 UI；禁止共享业务状态机、会话编排、请求编排。
6. 代码组织应能一眼识别双链路边界，保证后续可独立删除 NCP 或 Legacy 任一侧；但本方案阶段不预先锁死细分目录。

这里的执行原则不是“共享整个前端”，而是：

- 共享展示层
- 允许双编排层
- 统一 view-model 契约
- 入口统一切换

如果当前代码还无法清晰支撑这四条原则，需在 Phase 0 先做最小结构重排，再开始并行链路开发。

## 3.3 回滚原则

- 切换失败时，优先通过前端开关回到旧链路，保证业务连续性。
- 若阶段性判断新链路不可行，允许直接删除新链路代码，旧链路不受影响。
- 回滚动作必须无数据迁移依赖，不依赖手工修复存储。

## 4. 前置重构判定（先判定，再动手）

并行开发前先做一次“切换前置重构判定”，结论只有两类：

1. 不需要前置重构：
  - 现有前端入口已能稳定承载链路开关
  - 旧链路 API 与状态注入边界已足够清晰
2. 需要最小前置重构：
  - 当前入口层把 API 选择、状态编排、UI 容器混在一起，无法安全插拔
  - 需要先抽出“链路网关层/容器装配层”再接入新链路

前置重构只做“为切换服务”的最小改动，禁止借机做大规模风格重构。

## 5. 实施阶段

## Phase 0：切换基线与边界冻结

1. 固化旧链路基线能力清单（send/stream/abort/session list/session switch/session delete）。
2. 固化新链路 phase 1 的能力清单与非目标。
3. 定义切换开关位置、命名、默认值、可见性范围（dev/staging/prod）。

交付物：

- 链路能力对照表（Legacy vs NCP）
- 切换开关 ADR（放在 docs/designs 或 docs/plans）

## Phase 1：并行后端链路落地（存储不迁移）

1. 建立 NCP backend 主链路（session-centric）。
2. 通过 adapter 对接现有存储，保持会话/消息一致性。
3. 保留 legacy backend，确保两套路由均可工作。

交付物：

- NCP backend 路由与核心流程可运行
- 存储适配层说明文档（字段映射、语义映射、限制）

## Phase 2：并行前端链路落地（以切换点为中心）

1. 在前端入口接入链路开关。
2. 以 `@nextclaw/agent-chat-ui` 作为共享展示层基座，让 Legacy 与 NCP 共用 UI 组件、view-model 契约和 UI-local hooks。
3. 接入 NCP runtime + 容器链路，打通核心聊天主流程。
4. 保留 legacy frontend 容器路径，不做硬切。

交付物：

- 单一入口可切换 legacy/ncp
- Legacy 与 NCP 共用同一套展示层
- 核心流程在两条链路均可执行

## Phase 3：验证、切换、删旧

1. 在真实场景执行对比验证（稳定性、错误率、交互一致性）。
2. 满足阈值后将默认链路切到 NCP。
3. 进入旧链路删除阶段，按模块下线而非长期并存。

交付物：

- 切换与回滚记录
- 旧链路删除清单与完成状态

## 6. 验收标准（必须可验证）

## 6.1 功能验收

1. 两条链路都能完成核心主流程：发送、流式回复、停止、会话切换、会话删除。
2. 切换链路不要求用户迁移数据，已有会话数据可读。
3. 回滚到旧链路后功能与数据可用性不受影响。

## 6.2 架构验收

1. 新旧链路目录与核心模块清晰分离。
2. 新链路不反向依赖旧链路业务编排代码。
3. 旧链路可独立运行，新链路可独立删除。
4. 前端切换逻辑仅存在于入口装配层，展示层无链路判断分支污染。
5. `@nextclaw/agent-chat-ui` 继续保持共享展示层定位，不承载 runtime/store/presenter 级业务编排。

## 6.3 质量验收

1. NCP 相关包 lint/tsc/build 通过。
2. nextclaw-ui 与 nextclaw-server 受影响链路验证通过。
3. 新增并维护一条“链路切换冒烟用例”。

## 7. 风险与防护

1. 风险：双链路并行导致维护成本短期上升。
   - 防护：明确 phase 截止日期与删旧里程碑，禁止无限并存。
2. 风险：入口开关位置不当造成耦合扩散。
   - 防护：开关只允许存在于前端装配入口，不下沉到展示组件。
3. 风险：存储语义不一致导致“能读不能写”。
   - 防护：先定义 adapter 映射契约，再做实现与回归测试。

## 9. 执行口径

从本方案生效起，NCP phase 1 默认执行口径为：

1. 先并行，再切换，不在旧链路上做长期补丁式替换。
2. 先可回滚，再默认切换，不做不可逆改动。
3. 先删旧计划，再上新链路，确保最终收敛为单链路。
