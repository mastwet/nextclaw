# Incremental Maintainability Paydown

## 目标

把“降低技术债”从偶发的大型治理任务，变成每次正常开发都默认发生的一点点收敛动作。

这套机制不追求一次性还清历史债务，而是要求后续 AI 在完成主任务后，默认把被触达链路变得比开始时更干净一点、更清晰一点、更容易继续维护一点。

## 适用范围

默认适用于以下改动：

- 源码
- 脚本
- 测试
- 影响运行链路的配置

默认不适用于以下改动：

- 纯文档
- 纯措辞调整
- 元信息微调

## 默认原则

1. 主任务优先，但主任务完成不等于收工。
2. 每次代码任务默认都要评估一次“是否能顺手减一小笔债”。
3. 优先在同一文件、同一目录、同一模块、同一职责链路内减债，禁止借题发挥扩散到无关区域。
4. 优先删除和简化，再考虑新增抽象。
5. 每次只要求一个小而确定的减债动作，不要求顺手做大重构。
6. 如果没有安全、低风险、边界清晰的减债动作，必须明确写出原因与下一步拆分缝。

## 默认动作池

当 AI 完成主任务后，默认按以下顺序寻找一个可执行的 micro-paydown：

1. 删除一段死代码、重复分支、废弃 helper 或明显多余的中间层。
2. 把一段纯映射、纯归一化、纯格式化逻辑从大型编排函数中提取出来。
3. 用 guard clause 或小型 helper 收窄一个过深分支或长函数。
4. 修正一个已经被触达文件中的命名职责错配。
5. 把已经开始膨胀的目录补一个更明确的子目录边界，或补目录预算豁免说明。
6. 对被触达 hotspot 文件留下更清晰的拆分缝，而不是继续把逻辑塞进去。

这六类动作里，只要能安全完成其中一项，就算完成一次“增量减债”。

## 不应该做成什么样

以下做法不属于这套机制：

- 借“顺手减债”之名把任务范围扩成大重构
- 为了追求形式上的减债，新增一层没有必要的抽象
- 跨无关模块到处改，只为了让报告数字变好看
- 主任务本来很紧急，却因为顺手治理拖慢修复
- 明明没有做减债，却在结果里模糊表述成“顺便优化了一下”

## 何时必须更积极一点

如果本次触达以下区域，AI 应默认更积极地寻找一个 micro-paydown：

- maintainability hotspot 文件
- 目录预算进入 review / hard limit 的目录
- 当前 maintainability report 里的高违规文件
- 已经多次被触达但持续膨胀的编排层文件

这类场景不要求一次拆完，但要求至少别继续原样放大债务；如果能安全减一点，默认就应该减一点。

## 允许跳过的场景

只有在以下情况之一成立时，才可以不执行增量减债：

- 用户明确要求只做最小改动，禁止顺手优化
- 当前任务是高时效 hotfix，额外改动会提高风险
- 被触达区域虽然有债务，但本次没有安全、低风险、可验证的微小减债动作
- 减债动作会引入明显额外测试、迁移、发布或跨模块联动成本

即使跳过，也必须在结果里明确：

- 为什么这次不适合顺手减债
- 下一步最合理的拆分缝是什么

## 推荐执行顺序

1. 先完成用户主需求。
2. 再问三个问题：
   - 这里有什么能删掉？
   - 删不掉的部分有什么能简化？
   - 有没有一个低风险的小切口可以顺手拆出去？
3. 只选一个最划算、最小、最容易验证的动作执行。
4. 收尾时明确记录：
   - 本次是否顺手减债
   - 减掉了哪一笔债
   - 如果没减，为什么没减
   - 下一步拆分缝是什么

## 输出要求

未来 AI 在完成代码任务时，最终说明里默认应给出一行额外信息：

- `本次顺手减债：是/否`

若为 `是`，应写明减的是哪一笔。

若为 `否`，应写明原因和下一步拆分缝。

若本次还触达了 hotspot 文件，则还必须遵循 [`Maintainability Hotspot Freeze`](./maintainability-hotspot-freeze.md) 的留痕要求。

## 示例

### 示例 1：好例子

修复 `packages/nextclaw-core/src/agent/loop.ts` 某个分支 bug 时，顺手把消息归一化逻辑提取到独立 pure helper，并让 `processMessage` 少一个分支块。

这属于合格的增量减债，因为：

- 仍在同一文件/同一职责链路
- 风险边界清楚
- 主任务没有被拖慢
- 文件复杂度确实下降了一点

### 示例 2：也算合格

修改 `packages/nextclaw-ui/src/components/chat` 下某个组件时，发现目录已接近预算线，本次顺手把一个纯展示 helper 下沉到 `ui/` 子目录，避免继续堆在顶层。

### 示例 3：坏例子

本来只是修一个 provider 表单 bug，却顺手重写整个配置中心状态模型，导致任务风险和验证成本大幅上升。

这不是“增量减债”，而是失控扩 scope。
